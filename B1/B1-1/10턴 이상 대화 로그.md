1. 보고서 목적
- 작성 목적과 업무 배경을 바탕으로 1~2문장으로 작성한다.
- 제공된 시점, 대상 서비스, 주요 이슈가 있으면 반영한다.

2. 주요 목차
- 보고서 목적에 맞는 5개 목차를 번호 목록으로 작성한다.
- 각 목차에는 핵심 설명을 2~3줄 포함한다.
- 장애 분석 보고서는 장애 개요, 영향 범위, 원인 분석, 모니터링 검토, 재발 방지 중심으로 구성한다.
- 프로젝트 진행 보고서는 프로젝트 개요, 진행 현황, 주요 이슈, 일정 영향, 후속 조치 중심으로 구성한다.
- 성능 분석 보고서는 이슈 배경, 개선 범위, 적용 내용, 효과 검증, 운영 기준 중심으로 구성한다.

3. 핵심 메시지
- 보고서에서 강조할 핵심 메시지를 3개 작성한다.
- 입력된 사실에 근거하여 작성한다.

4. 예상 리스크
- “구분 / 리스크 / 고려사항” 형식의 표로 3개 작성한다.
- 근거가 부족한 내용은 “확인 필요”, “추가 분석 필요”, “가능성”으로 표현한다.

5. 개선 방향
- 개선 방향 또는 후속 조치를 3개 작성한다.
- 담당자, 완료 일정, 수치 목표는 입력되지 않은 경우 임의로 작성하지 않는다.

작성 규칙:
- 간결하고 객관적인 IT 회사 내부 보고서 문체를 사용한다.
- 단, 표현은 지나치게 딱딱하지 않게 하고 부드러운 협업형 톤을 유지한다.
- 입력되지 않은 사실, 수치, 일정, 원인, 정책, 담당자를 임의로 생성하지 않는다.
- 불확실한 내용은 “확인 필요”, “추가 분석 필요”, “가능성이 있다”로 표시한다.
- 정보가 부족하면 임의로 작성하지 말고 필요한 확인 질문을 제시한다.
- 보고서 목적과 관련 없는 내용은 포함하지 않는다.
- 개인정보, 계정정보, 내부 인증 정보 등 민감 정보는 직접 포함하지 않는다.

우선순위:
정확성 > 실무 활용성 > 간결함 > 부드러운 협업형 톤

모호한 입력 처리 기준

입력 정보가 부족하여 보고서 개요를 정확하게 작성하기 어려운 경우, 임의로 개요를 작성하지 않는다.

다음 항목을 중심으로 확인 질문을 제시한다.

1. 대상 서비스 확인 필요
- 문제가 발생한 시스템, 서비스, API 또는 기능명이 필요하다.
- 예: 로그인 API, 결제 시스템, 관리자 페이지, 배치 서버

2. 발생 시점 확인 필요
- 문제가 발생한 날짜와 시간대가 필요하다.
- 예: 2026년 6월 1일 10:00~11:00

3. 주요 증상 확인 필요
- “느림”, “문제 발생” 등 추상 표현의 구체적인 기준이 필요하다.
- 예: 평균 응답 시간 증가, 타임아웃 발생, 오류율 증가, 접속 지연

4. 영향 범위 확인 필요
- 전체 사용자 영향인지, 일부 사용자 또는 특정 기능에 한정된 문제인지 확인이 필요하다.

5. 원인 분석 자료 확인 필요
- 로그, 모니터링 지표, DB 부하, 서버 리소스 사용률 등 원인 판단에 필요한 자료 확인이 필요하다.

검색 API 성능 개선 결과 보고서 개요
보고서 목적
검색 API 응답 지연 개선 작업의 주요 내용을 정리하고, 개선 효과 검증 결과와 향후 운영 기준 수립에 참고하기 위함이다.
1.	성능 이슈 발생 배경
검색 조건이 복잡한 요청에서 검색 API 응답 지연이 발생한 것으로 확인되었다.
정확한 발생 시점, 영향 범위, 평균 응답 시간 변화는 추가 확인이 필요하다. 
2.	개선 작업 범위
2026년 4월 중 검색 API 성능 개선 작업이 진행되었다.
주요 개선 내용은 인덱스 추가와 쿼리 구조 개선이며, 개발팀, 운영팀, 서비스 기획팀 간 공유가 필요한 내용이다. 
3.	개선 적용 내용
복잡한 검색 조건 처리 시 DB 조회 성능을 개선하기 위해 인덱스를 추가하였다.
또한 쿼리 구조를 개선하여 검색 요청 처리 과정의 부하를 줄이는 방향으로 조정하였다. 
4.	개선 효과 검증 현황
개선 전후 성능 비교 수치는 현재 최종 검증 중이다.
따라서 개선 효과는 검증 결과가 확정된 이후 수치와 함께 정리하는 것이 적절하다. 
5.	향후 운영 기준 및 후속 조치
검색 API 응답 시간 기준, 복잡 조건 요청에 대한 모니터링 기준, 성능 저하 감지 기준을 정리할 필요가 있다.
최종 검증 결과를 바탕으로 추가 튜닝 또는 모니터링 기준 보완 여부를 검토할 수 있다.

•	개선 작업은 2026년 4월 중 진행되었으며, 성능 개선 효과는 현재 최종 검증 중이다. 
•	향후 안정적인 운영을 위해 성능 기준과 모니터링 기준을 함께 정비할 필요가 있다.
