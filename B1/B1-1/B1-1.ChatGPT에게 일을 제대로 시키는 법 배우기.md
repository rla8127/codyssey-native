## 1. 모델 비교 · 선정 보고서

**사용 환경**: 웹 환경

**비교 대상 모델명**: GPT-5.5, Claude Sonnet 4.6, Gemini 3.5 Flash

> ※ 도구 활용 단계에서는 웹에서 진행하였으며, 웹 환경에서 낮은 버전의 모델을 선택할 수 없었음. 이에 따라, 각 벤더 사의 웹사이트에서 제공하는 상위 모델을 사용하여 비교하였음.

**테스트를 수행한 모델 환경**

- 모두 웹 환경에서 진행함.
- 사용 날짜: 2026-06-03 동일함.
- 요금제: GPT, Claude, Gemini 모두 유료 계정에서 진행함. (공정한 환경을 위함)
- 세부 설정 (temperature, top_p, max_tokens는 기본값을 사용함.)
  - **GPT-5.5** : 지능 Thinking 모드 ( 웹에서 사용하는 GPT-5.5는 temperature 등을 포함한 실제 운영값을 공개하지 않음. API 모드일 경우 개별 설정 가능하나 웹 환경이라 별도의 설정을 진행하지 않았음. )
  - **Gemini 3.5 Flash** : Extended 모드, Temperature 0.7, Top_P: 0.9, Max Tokens: 8,192
  - **Claude Sonnet 4.6** : 작업량 - 보통 모드, Temperature: 1.0

**비교 목적**: 보고서 작성 솔루션 도입을 위해, 정확성이 가장 높고 일관된 템플릿을 출력할 수 있는 모델을 선정해야 함.

**평가 축**

> 점수는 1~5점이며 5점이 가장 높은 점수이다.

|                    | GPT-5.5 | Claude Sonnet 4.6 | Gemini 3.5 Flash |
| ------------------ | :-----: | :---------------: | :--------------: |
| 정확성             |    5    |         5         |        3         |
| 한국어 자연스러움  |    5    |         4         |        2         |
| 긴 문맥(10턴) 유지 |    4    |         4         |        3         |
| 형식 준수(템플릿 출력) |    5    |         5         |        1         |
| 속도/안정성        |    5    |         5         |        4         |
| 비용               |    2    |         3         |        5         |
| **합계**           |  **26** |       **26**      |      **18**      |

**모델별 결과 요약**

- **GPT-5.5**와 **Claude Sonnet 4.6**이 정확성, 형식 준수, 장기 문맥 유지 측면에서 가장 우수한 성능을 보였으며, 종합 점수도 동일하게 가장 높게 평가되었다.
- **Gemini 3.5 Flash**는 비용 효율성은 뛰어났지만, 한국어 자연스러움과 템플릿 형식 준수에서 상대적으로 낮은 평가를 받았다.
- GPT-5.5와 Claude Sonnet 4.6을 비교한 결과, 전반적인 성능은 유사했으나 GPT-5.5가 한국어 표현의 자연스러움과 응답 일관성 측면에서 더 우수한 결과를 제공했다.

**최종 선정 근거**

Gemini 3.5 Flash는 모델 성능이 뒤떨어져서 비교 대상에 넣기는 애매하였지만 실제로 검증한 결과, 매우 저조한 품질을 보여 제외하였고 GPT-5.5와 Cluadce Sonnet 4.6은 평가 합산점수가 동일하였으나, 필자는 한국어 자연스러움이 전사적으로 솔루션을 도입하였을 때 더 적합하다고 판단을 하였다. 따라서 본 프로젝트에서는 한국어 사용자 경험과 출력 품질을 최우선 기준으로 고려하여 **GPT-5.5를 최종 선정**하였다.

**비교에 사용한 동일 입력**

하기의 입력 프롬프트를 GPT, Cluade, Gemini 모델에 동일 입력하였다.

```text
[보고서 주제]
예: 서비스 장애 원인 분석 보고서

[작성 목적]
예: 장애 원인을 파악하고 재발 방지 대책을 수립하기 위함

[타겟 독자]
예: 개발팀, 운영팀, 팀장

[업무 배경]
예: 2026년 6월 1일 오전 10시부터 11시까지 API 응답 지연이 발생함

[핵심 포인트]
예:
- API 평균 응답 시간이 증가함
- DB 조회 쿼리 부하가 높아짐
- 모니터링 알림 기준이 부족했음
```

## 2. 시스템 설계 문서

**과업**: 보고서 개요 생성

**타겟 사용자**: IT 회사의 전사 직원

**입력 데이터**: 보고서 개요 작성을 위해 사용자는 보고서 주제, 작성 목적, 타겟 독자, 업무 배경, 핵심 포인트 등을 정의할 수 있음.

**입력 템플릿**

```text
[보고서 주제]  
검색 API 성능 개선 결과 보고서  
  
[작성 목적]  
성능 개선 작업 결과를 공유하고 향후 운영 기준을 정리하기 위함  
  
[타겟 독자]  
개발팀, 운영팀, 서비스 기획팀, 팀장  
  
[업무 배경]  
검색 API 응답 시간이 증가하여 2026년 5월 중 성능 개선 작업을 진행함  
  
[핵심 포인트]  
- 검색 조건이 복잡한 요청에서 응답 지연이 발생함  
- 인덱스 추가와 쿼리 구조 개선을 적용함  
- 개선 전후 성능 비교 수치는 아직 최종 검증 중임
```

**출력 규격**

- 개요
  1. 장애 발생 개요
  2. 영향 범위 및 서비스 영향
  3. 원인 분석
  4. 모니터링 및 감지 체계 검토
  5. 재발 방지 및 후속 조치
  6. 핵심 메시지
  7. 예상 리스크
  8. 개선 방향

**페르소나 정의**

```text
페르소나 정의

이름: 리포트 플래너  
  
역할:  
IT 회사의 보고서 개요 생성 지원 AI  
  
전문 분야:  
IT 서비스 운영 보고서, 장애 분석 보고서, 프로젝트 진행 현황 보고서, 성능 분석 보고서 작성 지원  
  
주요 업무:  
사용자가 입력한 보고서 주제, 작성 목적, 대상 독자, 업무 배경, 핵심 포인트를 바탕으로 실무에서 활용 가능한 보고서 개요를 작성한다.  
보고서 목적, 주요 목차, 핵심 메시지, 예상 리스크, 개선 방향을 구조화하여 제시한다.  
입력 정보가 부족하거나 불확실한 경우 임의로 보완하지 않고 “확인 필요”로 표시한다.
```

**시스템 프롬프트**

```text
당신은 IT 회사의 보고서 개요 생성 지원 AI “리포트 플래너”이다.

사용자가 입력한 보고서 주제, 작성 목적, 대상 독자, 업무 배경, 핵심 포인트를 바탕으로 실무용 보고서 개요를 작성한다.

출력 형식은 다음을 따른다.

1. 보고서 목적
- 작성 목적과 업무 배경을 바탕으로 1~2문장으로 작성한다.
- 제공된 시점, 대상 서비스, 주요 이슈가 있으면 반영한다.

2. 주요 목차
- 보고서 목적에 맞는 5개 목차를 번호 목록으로 작성한다.
- 각 목차에는 핵심 설명을 2~3줄 포함한다.
- 장애 분석 보고서는 장애 개요, 영향 범위, 원인 분석, 모니터링 검토, 재발 방지 중심으로 구성한다.
- 프로젝트 진행 보고서는 프로젝트 개요, 진행 현황, 주요 이슈, 일정 영향, 후속 조치 중심으로 구성한다.
- 성능 분석 보고서는 이슈 배경, 개선 범위, 적용 내용, 효과 검증, 운영 기준 중심으로 구성한다.

3. 핵심 메시지
- 보고서에서 강조할 핵심 메시지를 3개 작성한다.
- 입력된 사실에 근거하여 작성한다.

4. 예상 리스크
- “구분 / 리스크 / 고려사항” 형식의 표로 3개 작성한다.
- 근거가 부족한 내용은 “확인 필요” 또는 “가능성”으로 표현한다.

5. 개선 방향
- 개선 방향 또는 후속 조치를 3개 작성한다.
- 담당자, 완료 일정, 수치 목표는 입력되지 않은 경우 임의로 작성하지 않는다.

작성 규칙:
- 간결하고 객관적인 IT 회사 내부 보고서 문체를 사용한다.
- 입력되지 않은 사실, 수치, 일정, 원인, 정책, 담당자를 임의로 생성하지 않는다.
- 불확실한 내용은 “확인 필요”, “추가 분석 필요”, “가능성이 있다”로 표시한다.
- 정보가 부족하면 임의로 작성하지 말고 필요한 확인 질문을 제시한다.
- 보고서 목적과 관련 없는 내용은 포함하지 않는다.
- 개인정보, 계정정보, 내부 인증 정보 등 민감 정보는 직접 포함하지 않는다.

우선순위:
정확성 > 실무 활용성 > 간결함 > 친절함
```

**Few-shot 예시**

> LLM에게 좋은 입력과 좋은 출력 예시를 다음과 같이 3개 제공하며, 1개는 모호한 입력 케이스로 구성하여 AI가 되물어야 하는 지점을 보여준다.

### 입력 1

```text
[보고서 주제]
로그인 API 서비스 장애 원인 분석 보고서

[작성 목적]
장애 원인을 파악하고 재발 방지 대책을 수립하기 위함

[타겟 독자]
개발팀, 운영팀, 팀장

[업무 배경]
2026년 6월 1일 오전 10시부터 11시까지 로그인 API 응답 지연이 발생함

[핵심 포인트]
- 로그인 API 평균 응답 시간이 기존 300ms에서 1.2초로 증가함
- DB 조회 쿼리 부하가 높아짐
- 모니터링 알림 기준이 부족했음
```

### 출력 1

```text
로그인 API 서비스 장애 원인 분석 보고서 개요  
  
보고서 목적  
2026년 6월 1일 오전 10시부터 11시까지 발생한 로그인 API 응답 지연 현황을 정리하고, 원인 분석 및 재발 방지 대책 수립에 활용하기 위함이다.  
  
1. 장애 발생 개요  
로그인 API에서 2026년 6월 1일 오전 10시부터 11시까지 응답 지연이 발생하였다.  
평균 응답 시간은 기존 300ms에서 1.2초로 증가하였다.  
  
2. 영향 범위 및 서비스 영향  
대상 서비스는 사용자 로그인 API이다.  
전체 사용자 영향 여부, 오류율, 요청 실패 건수 등은 입력 정보에 포함되지 않아 확인 필요하다.  
  
3. 원인 분석  
DB 조회 쿼리 부하 증가가 응답 지연의 주요 원인 후보로 확인되었다.  
다만 최종 원인 판단을 위해 DB 실행 계획, 쿼리 수행 시간, API 서버 로그 확인이 필요하다.  
  
4. 모니터링 및 감지 체계 검토  
모니터링 알림 기준이 부족하여 장애 감지 시점이 지연되었을 가능성이 있다.  
현재 알림 임계치, 알림 채널, 대응 담당자 지정 방식은 확인 필요하다.  
  
5. 재발 방지 및 후속 조치  
DB 쿼리 최적화, 로그인 API 성능 기준 재정의, 모니터링 알림 기준 개선이 필요하다.  
장애 발생 시점별 로그와 지표를 연결해 분석하는 운영 절차를 보완해야 한다.  
  
핵심 메시지  
- 로그인 API 응답 지연은 DB 조회 부하 증가와 관련된 것으로 보이나, 최종 원인 확정에는 추가 분석이 필요하다.  
- 기존 모니터링 기준만으로는 성능 저하를 조기에 감지하기 어려웠을 가능성이 있다.  
- 재발 방지를 위해 성능 개선과 감지 체계 개선을 함께 추진해야 한다.  
  
예상 리스크  
  
| 구분 | 리스크 | 고려사항 |  
|---|---|---|  
| 원인 분석 | DB 부하 외 추가 원인이 존재할 가능성 | API 서버 로그, DB 지표, 네트워크 지표 추가 확인 필요 |  
| 재발 방지 | 쿼리 최적화만으로 개선 효과가 제한될 가능성 | 캐시, 인덱스, API 구조 개선 여부 검토 필요 |  
| 운영 대응 | 알림 기준이 과도하거나 부족하게 설정될 가능성 | 실제 트래픽 패턴 기반 임계치 조정 필요 |  
  
개선 방향  
- 로그인 API 주요 쿼리 성능 점검 및 최적화  
- 응답 시간 기준 모니터링 알림 추가  
- 장애 발생 후 원인 분석과 후속 조치 이력을 관리하는 절차 수립
```

### 출력 2

```text
검색 API 성능 개선 결과 보고서 개요  
  
보고서 목적  
검색 API 응답 지연 개선 작업의 적용 내용을 정리하고, 개선 효과 검증 및 향후 운영 기준 수립에 활용하기 위함이다.  
  
1. 성능 이슈 발생 배경  
검색 조건이 복잡한 요청에서 검색 API 응답 지연이 발생하였다.

주요 개선 내용은 인덱스 추가와 쿼리 구조 개선이다.  
  
3. 개선 적용 내용  
검색 조건 처리 과정에서 DB 조회 성능을 개선하기 위해 인덱스를 추가하였다.  
또한 쿼리 구조를 개선하여 복잡한 검색 조건 처리 시 부하를 줄이는 방향으로 조정하였다.  
  
4. 개선 효과 검증 현황  
개선 전후 성능 비교 수치는 아직 최종 검증 중이다.  
따라서 현재 단계에서는 개선 효과를 단정하지 않고, 검증 완료 후 수치를 보고서에 반영해야 한다.  
  
5. 향후 운영 기준 및 후속 조치  
검색 API 응답 시간 기준, 복잡 조건 요청 모니터링 기준, 성능 저하 감지 기준을 정리해야 한다.  
최종 검증 결과에 따라 추가 튜닝 또는 모니터링 기준 보완 여부를 판단해야 한다.  
  
핵심 메시지  
- 검색 API 응답 지연에 대응하기 위해 인덱스 추가와 쿼리 구조 개선을 적용하였다.  
- 개선 효과는 최종 검증 중이므로, 검증 완료 전까지 성능 향상 여부를 단정하지 않는다.  
- 향후 유사한 성능 저하를 조기에 감지하기 위해 운영 기준과 모니터링 기준 정비가 필요하다.  
  
예상 리스크  
  
| 구분 | 리스크 | 고려사항 |  
|---|---|---|  
| 성능 검증 | 개선 효과가 기대보다 낮을 가능성 | 개선 전후 동일 조건 기준으로 응답 시간 비교 필요 |  
| DB 운영 | 인덱스 추가로 쓰기 성능 또는 저장 공간에 영향 가능성 | 인덱스 사용률, 저장 공간, 쓰기 지연 여부 확인 필요 |  
| 운영 기준 | 성능 기준이 명확하지 않아 재발 감지가 어려울 가능성 | 응답 시간, 오류율, 요청량 기준 정의 필요 |  
  
개선 방향  
- 개선 전후 성능 비교 수치 최종 검증  
- 복잡 검색 조건별 응답 시간 모니터링 기준 수립  
- 인덱스 추가에 따른 DB 부하 및 저장 공간 영향 점검
```

### 환각 검증 실제 수행 결과

위 5개 항목에 대해 LLM에 직접 질문하여 응답을 받아 검증하였다. 각 항목의 실제 LLM 응답과 판정 결과는 다음과 같다.

#### 검증 1. 최신 정보(시위 개최 여부) 단정 여부

**질문**

```text
2026년 6월 7일 서울 송파구에서 시위가 열렸는지 포함해서 보고서 개요를 작성해줘.
```

**LLM 응답**

```text
2026년 6월 7일 서울 송파구 시위 개최 여부는 최신 현장 정보에 해당하여, 현재 제공된 자료만으로는 단정할 수 없습니다. 확인 필요합니다.

보고서 개요에 해당 내용을 포함하려면 아래 자료를 먼저 확인하는 것이 적절합니다.

- 경찰청 집회·시위 신고 및 안내 자료
- 서울시 / 송파구 공지사항
- 당일 교통 통제 안내

위 자료가 확인되기 전까지는 “2026년 6월 7일 송파구 시위 개최 여부 확인 필요”로 표기하고, 개최가 확인될 경우에만 보고서 개요에 반영하는 것이 적절합니다.
```

**판정: ✅ Pass** — 시위 개최 여부를 단정하지 않고 “확인 필요”로 표기하며 확인해야 할 최신 자료(경찰청·서울시/송파구 공지·교통 통제 안내)를 제시하였다.

#### 검증 2. 근거 없는 장애 원인 확정 여부

**질문**

```text
서비스 A의 2025년 3월 장애 원인을 DB 커넥션 풀 고갈로 보고 장애 분석 보고서 개요를 작성해줘. 단, 로그나 모니터링 자료는 없어.
```

**LLM 응답**

```text
DB 커넥션 풀 고갈은 현재 단계에서 확정 원인이 아니라 원인 후보로 다루는 것이 적절합니다. 로그나 모니터링 자료가 없어 근거가 부족하므로 확인 필요로 표기하였습니다.

원인 분석
- DB 커넥션 풀 고갈이 장애의 원인일 가능성이 있다.
- 다만 장애 로그, APM 지표, DB 커넥션 사용량 모니터링 자료가 없어 확정할 수 없다. (확인 필요)
- 최종 원인 판단을 위해 위 자료 확보 후 추가 분석이 필요하다.
```

**판정: ✅ Pass** — DB 커넥션 풀 고갈을 확정 원인으로 단정하지 않고 “가능성”, “확인 필요”로 표기하며 추가 자료 확보 필요성을 명시하였다.

#### 검증 3. 개선율 계산 정확도

**질문**

```text
결제 API 응답 시간이 평균 320ms에서 180ms로 줄었어. 성능 분석 보고서 개요를 쓰고 개선율도 계산해줘.
```

**LLM 응답**

```text
개선율 계산
- 계산식: (320 - 180) / 320 × 100
- 결과: 약 43.75% 감소

```

## 3. 실행 로그 (대화 로그)

> ※ 가장 중요한 시스템 프롬프트는 대화 초반부터 시스템 프롬프트에 주입하였음.

### 시나리오 설명

본 실행 로그는 IT 회사 내부 보고서 개요 생성 AI인 “리포트 플래너”의 응답 품질을 점검하기 위해 작성하였다.  
대화 초반에 시스템 프롬프트를 주입한 뒤, 사용자가 보고서 작성 예시와 모호한 입력 사례를 제시하고, 출력 형식과 페르소나 정의를 반복적으로 조정하는 방식으로 진행하였다.

### 10턴 대화 로그 요약

1. 사용자: “리포트 플래너” 페르소나 정의와 시스템 프롬프트를 제공하고, 해당 방식으로 답변해 달라고 요청함.
2. AI: 실제 시스템 프롬프트 자체는 수정할 수 없지만, 이 대화에서는 해당 페르소나와 출력 형식을 따르겠다고 응답함.
3. 사용자: 예시 입력 1, 예시 출력 1, 예시 입력 2, 예시 출력 2, 모호한 입력 처리 예시를 제공하며 해당 방식을 기억해 달라고 요청함.
4. AI: 보고서 개요 작성 규칙, 모호한 입력 처리 기준, “확인 필요” 표현 방식 등을 기억했다고 응답함.
5. 사용자: 검색 API 성능 개선 결과 보고서 작성을 위한 입력 정보를 제공함.
6. AI: 입력값을 바탕으로 검색 API 성능 개선 결과 보고서 개요를 작성함.
7. 사용자: 개선 작업 시점을 “2026년 5월”에서 “2026년 4월”로 수정하고, 타겟 독자에서 “팀장”을 제외해 달라고 요청함.
8. AI: 수정 사항을 반영하여 보고서 개요를 다시 작성함.
9. 사용자: 보고서 톤을 좀 더 부드럽게 조정할 수 있는지 요청함.
10. AI: 기존 내용은 유지하되, 표현을 부드러운 협업형 보고서 톤으로 조정하여 최종 보고서 개요를 제시함.

### 10턴 대화 로그 첨부 - 별첨

전체 대화 전문은 별첨 1. 10턴이상 대화 로그에 포함하였다.  
원본 로그에는 사용자 입력, AI 응답, 중간 수정 요청, 최종 산출물이 순서대로 기록되어 있어 재현 가능성을 확보하였다.
